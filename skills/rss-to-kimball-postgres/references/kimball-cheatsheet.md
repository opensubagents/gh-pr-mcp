# Kimball cheatsheet — for RSS-to-warehouse modeling

A focused reference. For the full treatment, see *The Data Warehouse Toolkit* (Kimball & Ross, 3rd ed.). This page is what you need to make decisions during an RSS modeling session.

## The vocabulary

**Fact table** — quantitative measurements of a business process at a declared *grain*. Long and skinny: many rows, few columns, mostly foreign keys + numeric measures. Grain is the most important thing about a fact table.

**Dimension table** — descriptive context: who, what, where, when. Wide and short relative to fact tables. Joined to facts by surrogate keys.

**Conformed dimension** — a dimension reused across multiple fact tables with the same meaning. `dim_date` is the canonical example. Conformed dims are how you compare facts across processes.

**Surrogate key** — integer (or bigint) generated by the warehouse, opaque, no business meaning. Lets you SCD2 a dimension without breaking facts. `GENERATED ALWAYS AS IDENTITY` in Postgres.

**Natural key** — the source-system identifier (RSS `guid`, an article URL, etc.). Stored as an attribute on the dim, *not* used as the join key from facts.

**Bridge table** — many-to-many between a fact and a dimension. E.g. one release announcement mentions multiple versions → `bridge_release_version`.

**Junk dimension** — flags and indicators that don't deserve their own dimensions, packed into one. For RSS this might collect `is_paid`, `has_video`, `is_advertorial` if such attributes existed.

**Degenerate dimension** — a source identifier that lives on the fact row itself, not in its own dim. RSS `guid` is often a degenerate dim if you don't need history (otherwise it goes on the SCD2 dim).

**Event-grained fact** — one row per occurrence of an event. RSS items are events.

**Periodic snapshot fact** — one row per entity per period (daily balance, weekly score). Use when you want to query "what did the feed look like on day X" cheaply.

**Accumulating snapshot fact** — one row per entity, updated as it moves through stages. Rare for RSS; useful if your "release" has a lifecycle (drafted → published → updated → retracted).

## The four-step design method (Kimball, ch. 1)

Apply this to *every* fact table:

1. **Identify the business process.** "Publish an RSS item." Not "store some data."
2. **Declare the grain.** Write it as one English sentence. "One row per RSS item per snapshot load." If you can't, stop and think harder before continuing.
3. **Identify the dimensions.** What describes the event? When (date), what (release), where (source feed), who (creator if present), how (category).
4. **Identify the facts.** Numeric, additive measures. `content_html_length`, `n_links`, `n_versions_in_range`.

## Slowly Changing Dimensions

Pick per dimension. Per *attribute* if you want to be precise.

| Type | Behavior | When to use |
|---|---|---|
| **SCD0** | Original value, never overwritten | Birthdate, original publication date |
| **SCD1** | Overwrite on change, no history | Cosmetic / corrective changes ("fixed a typo in the host"). `dim_link.host` |
| **SCD2** | New row per change, with `valid_from` / `valid_to` / `is_current` | Anything where history matters for queries. `dim_release.title`, `dim_release.content_len` |
| **SCD3** | Add a `prior_value` column | Single specific prior value matters. Rarely needed |
| **SCD4 (mini-dim)** | Move volatile attrs to a separate small dim | High-velocity attributes that bloat SCD2. Almost never needed for RSS |
| **SCD6 (1+2+3 hybrid)** | All three at once | When you genuinely need both current value lookup and history |

### SCD2 mechanics

```sql
CREATE TABLE IF NOT EXISTS dim_release (
  release_sk      bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  guid            text   NOT NULL,           -- natural key
  title           text   NOT NULL,
  link            text,
  valid_from      timestamptz NOT NULL,
  valid_to        timestamptz,                -- NULL or far-future for current
  is_current      boolean NOT NULL DEFAULT true,
  row_hash        bytea  NOT NULL,            -- digest of SCD2 attrs for change detection
  UNIQUE (guid, valid_from)
);
CREATE INDEX IF NOT EXISTS ix_dim_release_current
  ON dim_release (guid) WHERE is_current;
```

Update protocol on a change:
```sql
-- 1. Close the current row
UPDATE dim_release
   SET valid_to = now(), is_current = false
 WHERE guid = $1 AND is_current;

-- 2. Insert the new current row
INSERT INTO dim_release (guid, title, link, valid_from, row_hash)
VALUES ($1, $2, $3, now(), $4);
```

Detect changes by comparing `row_hash` (SHA-256 of canonicalized SCD2 attrs) before doing any UPDATE/INSERT. No-op if hash matches.

## Star vs. snowflake

Default to **star**: one fact, denormalized dims around it. RSS feeds are small; the snowflake's normalization savings don't matter, and joins multiply quickly.

Snowflake only when a dimension itself has a strict hierarchy you need to query independently (e.g. `dim_geography` exploded into city → region → country). RSS rarely calls for it.

## Bridge tables for many-to-many

When one fact relates to many dim members (one release mentions many links), use a bridge:

```sql
CREATE TABLE IF NOT EXISTS bridge_release_link (
  release_sk  bigint NOT NULL REFERENCES dim_release(release_sk),
  link_sk     bigint NOT NULL REFERENCES dim_link(link_sk),
  position    int    NOT NULL,        -- order in the source content
  link_text   text,
  PRIMARY KEY (release_sk, link_sk, position)
);
```

Bridges are weight-bearing; consider whether a "weight" or "allocation factor" is needed for additive aggregations (Kimball ch. 10).

## Event tables

Kimball calls these *transaction fact tables* in their event-grained form. For RSS, two natural events:

- **`event_feed_snapshot`** — every fetch of the feed: timestamp, feed `lastBuildDate`, item count, bytes.
- **`event_release_seen`** — every (snapshot, release) pair: was this release new in this snapshot, updated since the last, or unchanged?

These give you the audit log: "when did we first observe item X? when did its title change?"

## Don't model what you can't query

Test for every proposed table: name a query that needs it. If you can't, drop it. Simplicity in dimensional modeling is not a vice; it's the default. Add complexity when a question demands it, not because *The Toolkit* mentions a pattern.

## Anti-patterns to avoid

- **One-big-table.** Putting every RSS field on one wide table loses the joins that make star schemas queryable.
- **EAV (entity-attribute-value).** Tempting for "flexible" RSS, lethal for queries.
- **No grain comment.** Future you (or another modeler) will mis-grain a query within a week.
- **SCD2 everywhere.** Bloats history with no payoff.
- **Surrogate key = natural key.** Loses SCD2 capability forever.
- **Foreign keys missing.** "We'll add them later" is how warehouses rot.
